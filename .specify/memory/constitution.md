# Conduit Constitution

## Core Principles

### I. Agent-First Development

Every layer of the development workflow MUST be discoverable and
invocable by AI coding agents without human hand-holding.

- `CLAUDE.md` at the repository root provides agents with full project
  context: tech stack, folder structure, build commands, critical rules,
  and conventions
- Each Nuke build target MUST have a corresponding skill definition in
  `.claude/skills/`, auto-generated and kept in sync via
  `nuke LintSkillsVerify`
- Domain-specific coding rules MUST be maintained in `.claude/rules/`
  organized by area (backend, frontend, e2e, functional-tests, cicd)
- Agents MUST be able to self-diagnose failures by inspecting `Logs/`
  and `Reports/` artifacts — Nuke targets MUST emit guidance pointing
  to these outputs on failure

### II. Clean Architecture with CQRS

The backend MUST follow Ardalis Clean Architecture with strict
separation of concerns and CQRS via MediatR.

- Endpoints MUST be thin: bind request, authorize, delegate to MediatR
  command/query, map response
- Business rules MUST live in MediatR handlers, never in endpoints
- All handlers MUST return `Result<T>` with typed status codes — no
  throwing exceptions for control flow
- Persistence MUST be abstracted behind repository interfaces —
  handlers MUST NOT reference EF Core directly
- Cross-cutting concerns (transactions, exception handling, logging)
  MUST be handled by MediatR pipeline behaviors, not repeated in
  handlers

### III. Compile-Time Guardrails (NON-NEGOTIABLE)

Custom Roslyn analyzers MUST enforce architecture, persistence,
endpoint, and testing conventions at compile time. These guardrails
exist so that code — whether written by humans or agents — is
structurally correct before it ever runs.

- Roslyn analyzers and ArchUnit rules MUST NOT be modified unless
  explicitly instructed by a human
- Warnings and errors MUST NOT be suppressed in code unless explicitly
  instructed
- New patterns MUST be enforced via analyzers when they represent
  invariants that affect correctness

### IV. Nuke as Single Entry Point

All build, test, lint, migration, and run operations MUST go through
`nuke <Target>`. Direct invocation of `dotnet`, `npm`, or `docker`
commands is prohibited.

- Every Nuke target MUST be documented and have a corresponding Claude
  Code skill
- If modifying the Nuke build, it MUST build successfully before
  committing
- Nuke target failure messages MUST include specific guidance on where
  to find logs and reports

### V. 4-Layer Test Suite

The project MUST maintain four complementary test layers covering
different scopes and concerns.

- **xUnit** — backend business logic, handlers, persistence
  (`App/Server/tests/`)
- **Vitest + Testing Library** — frontend components, hooks, utilities
  (`App/Client/`)
- **Playwright** — full user flows through the browser (`Test/e2e/`),
  using Page Object Model, ARIA selectors, and `Expect()` assertions
- **Postman/Newman** — API contract validation (`Test/Postman/`)
- All test results MUST be output to `Reports/` with HTML reports
  generated by Nuke
- Postman tests MUST pass before finishing any API work

### VI. Observability

Runtime behavior MUST be observable through structured logs and audit
trails written to well-known directories on disk.

- **Serilog** — structured logging with properties, enrichers, and file
  sinks in `Logs/`
- **Audit.NET** — automatic audit trails for EF Core entity changes and
  Identity operations in `Logs/`
- Logs MUST use structured properties (`{@Command}`, `{UserId}`,
  `{Slug}`) — never log secrets or JWTs
- Playwright traces MUST be automatically captured on E2E test failure
  for agent-driven diagnosis

### VII. No Magic, No Noise

Code MUST be explicit, minimal, and free of ceremony that adds no
value.

- Magic strings are prohibited — use configurable values, enums,
  constants, or reflection (exceptions: SQL strings, UI text)
- XML documentation comments are prohibited unless explicitly requested
- Code comments are prohibited unless the logic is inherently unclear —
  preserve existing comments
- Mass refactoring via scripting tools (python, perl, awk, sed, regex)
  is prohibited — only direct, manual edits

## Technology & Architecture Constraints

- **Backend**: .NET 10, FastEndpoints, MediatR, FluentValidation,
  EF Core + SQLite, Serilog, Audit.NET, Microsoft.FeatureManagement
- **Frontend**: React 19, Vite, TypeScript, Carbon Design System
- **Build**: Nuke Build, GitHub Actions CI/CD
- **Infrastructure**: Docker, Bicep (Azure), Azure App Service
- **API Client**: Kiota-generated TypeScript client — regenerate via
  `nuke BuildGenerateApiClient`, drift-checked via
  `nuke LintApiClientVerify`
- **Architecture**: Ardalis Clean Architecture (no Aspire) with
  semantic folder structure (`App/`, `Task/`, `Test/`, `Infra/`,
  `Logs/`, `Reports/`)
- GitHub Actions job names MUST match kebab-case Nuke target names
- Path-based job gating with `dorny/paths-filter@v3`

## Development Workflow & Quality Gates

- `nuke LintAllVerify` MUST pass before any commit
- `nuke LintSkillsVerify` MUST confirm all Nuke targets have
  corresponding Claude Code skills
- `nuke LintApiClientVerify` MUST confirm the generated TypeScript
  client is in sync with backend endpoints
- All mutating API requests MUST have a FluentValidation validator
  returning 422 with a consistent problem/details payload
- Frontend API modules MUST wrap Kiota path builders, cast responses
  to app types, and catch errors via `convertKiotaError()`
- E2E tests MUST use Page Object Model, ARIA selectors, and
  `Expect()` assertions — never `WaitForAsync()`, `Task.Delay()`,
  or brittle CSS selectors
- Backend functional tests MUST use FastEndpoints test extensions,
  never raw `HttpClient`
- Commits MUST NOT include documentation updates unless explicitly
  requested

## Governance

This constitution supersedes all other development practices for the
Conduit project. Amendments require:

1. A description of the change and its rationale
2. Verification that the change does not violate existing Roslyn
   analyzer rules or break the test suite
3. An update to `CLAUDE.md` if the amendment affects agent-facing
   instructions
4. A version bump following semantic versioning:
   - **MAJOR**: Principle removal or backward-incompatible redefinition
   - **MINOR**: New principle or materially expanded guidance
   - **PATCH**: Clarifications, wording, or non-semantic refinements

All PRs and code reviews MUST verify compliance with these principles.
Complexity beyond what is prescribed here MUST be explicitly justified.

Use `CLAUDE.md` and `.claude/rules/` for runtime development guidance.
