## ЁЯПЫя╕П Architecture

The backend follows [Ardalis Clean Architecture](https://github.com/ardalis/CleanArchitecture) with CQRS via MediatR:

```
Server.Core              тАФ Domain models, aggregates, value objects
Server.SharedKernel      тАФ Common types and shared abstractions
Server.UseCases          тАФ MediatR command/query handlers (business logic)
Server.Infrastructure    тАФ EF Core DbContext, repositories, persistence gateways
Server.Web               тАФ FastEndpoints endpoints, DTOs, mappers, middleware
Server.Web.DevOnly       тАФ Development-only endpoints (seeding, diagnostics)
Server.Analyzers         тАФ 32 custom Roslyn analyzers (SRV + PV series)
```

Endpoints are thin: bind request тЖТ authorize тЖТ delegate to MediatR тЖТ map response. Business rules live in handlers. Persistence is abstracted behind repository interfaces.

**Cross-cutting concerns:**
- **Serilog** тАФ structured logging with enrichers, file sinks, and Seq integration
- **Audit.NET** тАФ automatic audit trails for EF Core entity changes and Identity operations
- **Microsoft.FeatureManagement** тАФ feature flags for toggling functionality at runtime

## ЁЯдЦ Agent-First Workflow

This template is designed to be operated by AI coding agents. Every layer of the development workflow is exposed in a way that agents can discover and invoke.

### ЁЯУВ Semantic Folder Structure

The repository's top-level directories provide clean semantic separation that agents can navigate without guesswork:

- **`App/`** тАФ all application source code (frontend and backend)
- **`Task/`** тАФ build system and local dev tooling (Nuke, Docker Compose)
- **`Test/`** тАФ integration and contract tests (Playwright, Postman, migration verification)
- **`Infra/`** тАФ infrastructure as code (placeholder)
- **`Logs/`** тАФ runtime logs (Serilog, Audit.NET)
- **`Reports/`** тАФ generated test reports

Each directory has a single, clear responsibility. An agent can immediately locate where application code lives (`App/`), how to build it (`Task/`), how to test it (`Test/`), and where to find runtime output (`Logs/`, `Reports/`). No ambiguity, no overlapping concerns.

### ЁЯУЛ CLAUDE.md тАФ Project Instructions

The `CLAUDE.md` file at the repository root provides agents with full project context: tech stack, folder structure, build commands, critical rules, and conventions. This is the entry point for any agent working on the codebase.

### ЁЯОп `.claude/skills/` тАФ 46 Build Skills

Each Nuke build target has a corresponding skill definition in `.claude/skills/`. These are auto-generated and kept in sync via `nuke LintSkillsVerify`. Agents can invoke any build operation by name:

- `nuke-build-server` тАФ compile the .NET backend
- `nuke-test-e2e` тАФ run Playwright end-to-end tests
- `nuke-lint-all-fix` тАФ auto-fix all lint issues
- `nuke-db-migrations-add` тАФ add a new EF Core migration
- `github-push-pr` тАФ push to PR and monitor CI, auto-investigating failures

Skills also include workflow skills like `debug` (structured debug analysis) and `on-stop` (pre-commit verification).

### ЁЯУП `.claude/rules/` тАФ Coding Rules

Domain-specific coding rules for agents organized by area:

- `backend.md` тАФ endpoint patterns, CQRS conventions, persistence rules, error handling
- `frontend.md` тАФ React/TypeScript conventions, Carbon Design System usage
- `e2e.md` тАФ Playwright Page Object Model, ARIA selectors, assertion patterns

### ЁЯЫбя╕П Compile-Time Guardrails тАФ Roslyn Analyzers + FastEndpoints Framework

The backend is built on a custom framework layered on top of FastEndpoints, `Result<T>`, and MediatR pipeline behaviors. This framework constrains generated code to be clean and correct by default тАФ and 32 custom Roslyn analyzers enforce it at compile time.

**The framework:**
- **`Result<T>`** тАФ all handlers return `Result<T>` with typed status codes (Ok, NotFound, Invalid, Conflict, CriticalError, etc.) instead of throwing exceptions. This makes error handling explicit and composable
- **`ICommand<T>` / `IQuery<T>`** тАФ CQRS interfaces that expose `T` at compile time via `IResultRequest<T>`, enabling type-safe pipeline behaviors without reflection
- **MediatR pipeline behaviors** тАФ every request flows through `ExceptionHandlingBehavior` (catches exceptions тЖТ Result), `TransactionBehavior` (wraps commands in DB transactions, auto-commits on success, rolls back on failure), and `LoggingBehavior` (structured logging with timing)
- **`Send.ResultMapperAsync`** тАФ FastEndpoints extension that maps `Result<T>` status to HTTP responses automatically (Ok тЖТ 200, NotFound тЖТ 404, Invalid тЖТ 422, etc.)

The result: endpoints are thin wrappers (`request тЖТ mediator тЖТ result тЖТ response`), business logic lives in handlers, and cross-cutting concerns (transactions, error handling, logging) are handled by the pipeline тАФ all enforced at compile time by 32 custom Roslyn analyzers covering architecture boundaries, persistence patterns, endpoint conventions, and testing rules.

### ЁЯУК Observability тАФ Logs & Reports on Disk

Application logs and test results are written to well-known directories on disk, giving agents concrete artifacts to inspect when something goes wrong:

- **`Logs/`** тАФ Serilog structured logs and Audit.NET audit logs, organized by run mode
- **`Reports/`** тАФ HTML test reports from xUnit, Vitest, and Playwright runs

Nuke build targets emit explicit guidance on failure, pointing agents to the exact log files and reports to inspect. Claude Code hooks (`.claude/hooks/`) enforce that agents read these outputs rather than guessing at root causes тАФ creating a closed feedback loop between running a command and diagnosing its result.

### ЁЯФО Playwright Trace Debugging

When Playwright E2E tests fail, traces are automatically captured to disk. The `view-playwright-traces` skill lets agents analyze these trace files to see exactly what happened тАФ page navigations, network requests, screenshots, and DOM snapshots at each step. This means agents can self-diagnose E2E failures without human intervention: run the tests, inspect the trace, identify the issue, and fix it.

### ЁЯФД Kiota API Client Generation

When backend endpoints change, run `nuke BuildGenerateApiClient` to regenerate the TypeScript API client. The drift check `nuke LintApiClientVerify` catches when the generated client is out of sync. This means agents adding or modifying API endpoints get frontend bindings automatically тАФ no manual sync required.
